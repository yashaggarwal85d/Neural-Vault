## Text Elements
Functional requirements - 

PUT object (key → blob)
GET object
DELETE object
LIST objects (prefix-based)
Metadata per object (size, checksum, version)
Multipart upload (large files) ^rGcpCb48

Non-Functional Requirements

1. Availability
    99.99%+ availability
    System must continue serving reads during node failures
2. Durability
    No data loss (e.g., 11 nines durability like S3)
    Data replicated across failure domains
3. Consistency
    Eventual consistency for reads after write
    Strong consistency for metadata operations (optional tradeoff)
4. Scalability
    Horizontally scalable
    No single point of failure
    Metadata must scale independently
5. Performance
    Read-heavy system
    Low latency reads (<100ms P99)
6. Fault Tolerance
    Node, rack, AZ failures
    Automatic re-replication
7. Security
    Authentication & authorization
    Per-object ACLs / IAM roles ^X0Ep4jph

Scale - 

Writes: 1M writes/sec
Reads: 100M reads/sec
Avg object size: 1MB
Storage per day ≈ 1PB/day

Write flow - 

1. Client → Coordinator
    Client sends PUT(key, object)
    Request can hit any node
    That node becomes coordinator
2. Replica Selection
    Coordinator hashes the key using consistent hashing
    Identifies:
        Primary replica
        N-1 secondary replicas
    Replicas are chosen across failure domains
3. Parallel Write to Replicas
    Coordinator sends write requests in parallel to all replicas
    Each replica:
        Writes data to disk (append-only)
        Writes metadata (version, checksum)
        fsync / WAL for durability
        Returns ACK
4. Quorum Acknowledgement
    Coordinator waits for W successful ACKs
        If W ACKs received: Return success to client
        If < W ACKs: Return failure
    Ensures availability + durability tradeoff
5. Failure Handling
    If some replicas fail:
        Mark them as hinted handoff candidates
        Store temporary hints
    Background processes:
        Anti-entropy (Merkle trees)
        Read-repair during GETs
        Replica rebuild
6. Versioning & Conflict Resolution
    Each object has:
        Version ID / vector clock
    Concurrent writes:
        Last-write-wins (timestamp)
        Or versioned objects (S3-style)
7. Coordinator Failure
    If coordinator crashes mid-write:
        Client retries with same request ID
        Writes are idempotent
        Partial writes resolved via quorum & repair ^7Mc7qdvQ
