## Text Elements
Functional Requirements

1. Key-value operations:
    GET(key)
    SET(key, value, ttl)
    UPDATE(key, value)
    DELETE(key)
2. Values are strings; keys are strings
3. Per-key TTL support
4. Automatic eviction using:
    LRU (default)
    LFU (configurable)
5. Single-key operations only
    (No transactions, no conditional writes, no multi-key ops) ^Qe4byd7W

Non-Functional Requirements
    
1. High availability: 99.99%
2. Very low latency:
    Reads: <1ms P99
    Writes: <1ms P99
3. Horizontal scalability
    Add/remove nodes dynamically
    Minimal rebalancing
    Replication for availability, not durability
4. Eventual consistency is acceptable
5. Cache data loss is acceptable on catastrophic failures ^n9vC6fKm

Scale Assumptions

1. Writes: 1M ops/sec
2. Reads: 5M ops/sec
3. Average value size: ~1 KB
4. Total data size: ~TBs (distributed across nodes) ^9BYxt8i7

Misc

1. Conflict resolution - 
    Each value stores Version timestamp
    Last-write-wins (LWW) strategy
2. Replica Placement - 
    For replication factor N
    After primary virtual node:
        Next N−1 distinct physical nodes clockwise become replicas
3. Hinted Handoff 
    Coordinator temporarily stores write for failed node
    Replays when node comes back
4. Eviction
    Happens with internal Algorithms
    LRU use doubly linked list
 ^45gC8d1e

Write Flow

1. Client sends SET(key, value, ttl) to any node
2. Node becomes coordinator
3. Coordinator:
    Computes responsible replica nodes
    Sends write requests in parallel
4. Write succeeds after W acknowledgements
5. Remaining replicas updated asynchronously
6. If a replica is down:
    Write is skipped
    Repaired later via anti-entropy ^jh99j3fX

Read Flow

1. Client sends GET(key) to any node
2. Node becomes coordinator
3. Coordinator:
    Reads from R replicas
    Returns latest version to client
4. Triggers read repair if inconsistencies detected ^WpTgsvAQ
