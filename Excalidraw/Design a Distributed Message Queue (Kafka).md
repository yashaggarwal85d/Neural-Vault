## Text Elements
Functional requirements - 

1. Publish messages to a topic
2. Topics should be Logically divided into partitions
3. Consumers should be able to:
    Subscribe to topics
    Consume messages in order per partition
4. Support consumer groups
    Each partition consumed by only one consumer in a group
5. Support offset management
    Consumers can replay messages
6. Support retention policies
    Time-based
    Size-based
7. Support delivery guarantees
    At-least-once (minimum)
    Exactly-once (optional, advanced)
8. Support source & sink connectors (Kafka Connect–like) ^Lv3Y0xvP

Non-functional requirements - 

1. High throughput : Millions of messages/sec
2. Low latency : Single-digit milliseconds (producer → broker)
3. High availability : No data loss on broker failure
4. Durability : Messages persisted to disk
5. Horizontal scalability : Add brokers without downtime
6. Fault tolerance : Broker, disk, network failures
7. Ordering guarantees : Strict ordering within a partition
8. Multi-tenancy : Isolation across topics/tenants ^Y9zw8S9P

Scale - 
~ 1M topics
~ 100M messages/sec ingestion
Message size: 1 KB (avg)
Read traffic ≥ write traffic
Cluster size: 100s–1000s of brokers ^hNKpOvsJ

Producer flow
Metadata → Partition selection → Batch → Leader write → Replication → ACKs

Consumer flow
Group join → Partition assignment → Pull data → Process → Commit offset

Failure handling
Broker failure → ISR leader election → Metadata refresh → Resume traffic
