# Excalidraw Data

## Text Elements
Functional requirements - 

* Operations - 
    1. Put
        put(key, value)
        Overwrites existing value
        Supports large values (blob / JSON)
    2. Get
        get(key)
        Fetch value by primary key
    3. Delete
        delete(key)
    4. Conditional Writes (CAS)
        put(key, value, if_version = X)
        Prevents lost updates
        Used for concurrency control

* Data Model
    1. Primary key only (no joins)
    2. Key: string / integer
    3. Value: opaque blob (JSON / binary)
    4. Each item has:
        Version / timestamp
        Metadata (checksum, TTL optional)

* Query Model
    1. Key-based access only
    2. No full table scans
    3. No joins
    4. No complex filtering
    5. Optional secondary indexes (out of scope for core design) ^McWY4PIC

Non functional requirements - 

* Availability & Fault Tolerance
    1. 99.99% uptime
    2. System must tolerate:
        Single node failure
        Multiple node failures
        Rack / AZ failure
    3. No data loss on failures

* Scalability
    1. Horizontally scalable
    2. Add/remove nodes with minimal rebalancing
    3. Automatic partitioning and rebalancing

* Performance
    1. Low latency
        Reads: ~5–10 ms (p99)
        Writes: ~10–20 ms (p99)
    2. High throughput
    3. Consistency
        Eventual consistency by default
        Strong consistency (opt-in) within a region
    4. Cross-region replication is eventual

* Data Durability
    1. Data must be persisted to disk
    2. Replicated across multiple nodes
    3. Replication factor ≥ 3

* Traffic Patterns
    1. Read-heavy workload (≈ 5:1 read/write)
    2. Skewed access possible (hot keys)
    3. Bursty traffic expected

* Hot Key Handling
    1. Virtual nodes
            split hot key
                user123 → user123#1, user123#2 ...
    2. Adaptive partitioning
    3. Throttling - Rate limit abusive keys ^VvvBixAQ

Scale Assumptions - 

* Traffic
    Writes: 1 million / second
    Reads: 5 million / second

* Data
    Hundreds of TB to Petabytes
    Average value size: ~1 KB
    Max value size: configurable (e.g., 400 KB like DynamoDB)

* Cluster
    Thousands of nodes
    Multi-AZ deployment
    Single region (for now) ^6n2G7Cdh

Write Path - 

1. Client sends write to any node
2. That node becomes the coordinator
3. Coordinator:
    Hashes key
    Finds N replica nodes using consistent hashing
4. Coordinator sends write request to all N replicas
5. Each replica:
    Writes data to memory (memtable)
    Appends to WAL (Write-Ahead Log) (for durability)
    Updates version/vector clock
6. Coordinator waits for W acknowledgements
7. Once W acks received:
    Responds success to client
    Remaining replicas may finish asynchronously
8. If total less than W acks recieved:
    write fails
    Partial writes may still exists 
    Eventually overwritten or repaired
9. Hinted Handoff :
    Used when replica is temporarily down
    Missed writes are stored as hints
    Delivered when node recovers ^RmAi7Iqj

Read path - 

1. Client sends read to any node
2. Node becomes coordinator
3. Coordinator:
    Finds N replicas
4. Sends read requests to R replicas
5. Receives responses
6. Chooses latest version using vector clocks
7. Returns value to client
8. If stale replicas detected:
    Performs read repair asynchronously

 ^s5T4Yf2q

Misc

1. Anti-Entropy - 
    Periodic background sync
    Uses Merkle trees
    Compares replicas and Repairs missing data
2. Gossip
    Highly available eventual consistent
    Zookeper is strongly consistent with central dependency
