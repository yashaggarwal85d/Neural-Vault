# Excalidraw Data

## Text Elements
Functional requirements -

1. Seed-based Crawling: 
    Start with a set of URLs and discover new ones by extracting links.
2. Politeness: 
    Adhere to robots.txt and implement per-host rate limiting to avoid DDoS-ing small servers.
3. Content Extraction: 
    Store the HTML content in a blob store and metadata (title, snippet) in a DB.
4. Deduplication: 
    Use Bloom filters or Checksums to avoid processing the same content/URL twice. ^RKAv42zz

Non functional requirements -

1. Scalability: 
    Must handle billions of pages using a distributed worker pool.
2. Extensibility: 
    Must be easy to add new modules (e.g., a module to crawl images or a module to handle JavaScript rendering).
3. Robustness: 
    Handle "spider traps" (infinite loops), malformed HTML, and dead links gracefully. ^qOPnj726

Scale - 

1. Throughput: Assume 5 Billion pages per month.
2. QPS: 5x10^9/ (30 days x 86400seconds)  = approx 2,000 pages per second.
3. Storage (100KB/page): 5B x 100KB = 500TB per month. Over 5 years, that's 30 PB. ^vmeWvm7g

API design - 

1. POST /v1/jobs: Starts a crawl task. Returns a job_id.
2. GET /v1/jobs/{id}: Returns status (e.g., QUEUED, IN_PROGRESS, COMPLETED, FAILED).
3. GET /v1/pages/{url_hash}: Retrieves metadata and storage pointers for a specific page. ^m5nZRVDk
