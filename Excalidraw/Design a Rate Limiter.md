## Text Elements
Functional requirements -

1. Rate limit per user ID, fallback to IP
2. Enforce multi-level limits:
    per second
    per minute
    per day
3. On breach:
    Return HTTP 429 Too Many Requests
4. Include retry-after metadata
5. Limits are configurable (free / premium / admin)
6. System should allow small burstss ^NTQdqYfr

Non - Functional requirements -

1. Very low latency (<5 ms ideally)
2. Highly available
3. Fail-open if rate limiter is unavailable
4. Horizontally scalable
5. Read-heavy 
6. place the rate limiter as close to the request entry point as possible
    At the API Gateway / Edge
    Or as a library embedded in application servers, backed by a shared store like Redis
    No synchronous call to a separate rate-limiter service

 ^S7qKfqEP

Scale - 

1. 10M users
2. 1M DAU
3. Average QPS ≈ 1–2k
4. Peak QPS: 10–20x → 20k–30k QPS
 ^ZvFgvAzR

Popular Rate Limiting Algorithms -

1. Fixed Window Counter
    Counts requests in a fixed time window (e.g., per minute); 
    simple but allows bursts at window boundaries.
2. Sliding Window Log
    Stores timestamps of each request and checks the last N seconds
    accurate but very memory-heavy.
3. Sliding Window Counter
    Approximation of sliding window using multiple counters; 
    better accuracy than fixed window with less memory.
4. Token Bucket
    Tokens refill at a fixed rate; requests consume tokens; 
    allows controlled bursts (most commonly used and preferred choice). 
    Refill is calculated lazily on request avoiding background jobs and scale better
    Stores - user/ip, time_window, capacity, token, last_refill_timestamp
5. Leaky Bucket
    Requests exit at a constant rate like water leaking; 
    smooths traffic but does not allow bursts.

 ^yNkBt8tP

Request flow - 

1. Identify user (or IP)
2. Fetch all required buckets (sec, min, day)
3. For each bucket:
    Recalculate tokens based on elapsed time and rules
    Single Redis Lua Script call (industry-preferred)
    No race conditions if multiple servers
    Atomicity guaranteed
    If tokens < 1 → reject request
4. If all buckets pass:
    Decrement 1 token from each bucket
    Allow request
5. On reject:
    Return HTTP 429
    Include Retry-After ^iOV72Qb9
