## Text Elements
Functional requirements - 

1. Users can:
    Upload, download, delete files
    Organize files into folders
2. Files should:
    Sync automatically across multiple devices
    Support versioning and rollback
3. Users can:
    Share files/folders with other users
    Generate public or private shareable links
    Revoke access at any time
4. System should:
    Handle large files (GBs)
    Resume interrupted uploads/downloads ^1nKksP0C

Non-Functional Requirements - 

1. High Availability : File access should survive node/zone failures
2. Strong Durability : Once uploaded, files must never be lost
3. Scalability : Horizontally scale to billions of files and users
4. Eventual consistency
5. Low Latency : Metadata operations < 100ms
6. Fault Tolerance : No single point of failure
7. Security
    Authentication & authorization
    Encryption at rest & in transit
    Malware scanning
    Rate Limiting & Abuse Prevention
        Per user / per IP
8. Cold storage for infrequently accessed files
 ^Zd5bwxec

Scale - 

1. 1M daily active users
2. 10M file uploads/day
    Avg ~120 uploads/sec
    Peak ~1k uploads/sec
3. 500M file reads/day
    Avg ~6k reads/sec
    Peak ~50k reads/sec
4. Average file size ~5–10MB
5. Read-heavy system (~95% reads)
6. Storage grows by tens of TB per day
    10M uploads/day × 5MB ≈ 50TB/day
    18PB/year (before replication) ^62n1AXjD

Conflict resolution

Each device syncs changes along with the last known file version.
The metadata service compares this with the current version.
If the versions differ, the system detects a conflict.
For binary files, we create a conflicted copy instead of overwriting.
This guarantees no data loss while allowing the user to resolve conflicts ^1qS6G5HS
