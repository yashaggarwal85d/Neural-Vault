## Text Elements
Functional requirements - 

1. Given a key, deterministically map it to a responsible node
2. Support adding and removing nodes with minimal key remapping
3. Ensure even key distribution across nodes
4. Support virtual nodes for load balancing
5. Allow independent clients to compute routing without central coordination ^YV3ZUmZS

Non functional requirements - 

1. High availability — routing should work even during node churn
2. Horizontal scalability — thousands of nodes
3. Low latency — O(log N) or better lookup
4. Minimal data movement on membership changes
5. Eventual consistency for ring membership across clients
6. Fault tolerance — node failures should not break routing
Routing is pure computation, not storage.
Replication Strategy : 
    Store a key on the next R distinct nodes clockwise on the ring ^v3tRLa76

Scale - 

1. Thousands of nodes
2. 100k requests/second
3. Nodes may fail/scale up/down
Temporary inconsistencies are expected ^MmNKpHDB

Virtual nodes - 

Say we have:
    10 physical nodes
    Hash space: 0 → 2³²
    Each node is placed once on the ring.
Because hashing is random:
    Some nodes may get large gaps
    Some may get tiny gaps
Result:
    One node may receive 30–40% of keys
    Another may receive 2–3%

Instead of placing a node once:
    Each physical node is placed multiple times
    Each placement is a virtual node (vnode)
    Example:
        50–200 virtual nodes per physical node

When a node fails:
    All of its virtual nodes are removed from the ring
    Each affected key:
        Moves clockwise to the next virtual node
    Only keys belonging to the failed node move
    No global reshuffle
    Some clients still think the node is alive:
        Client-side retry, then routes to the next node
     ^HernSKsD
